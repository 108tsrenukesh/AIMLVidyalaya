Load & EDA df = pd.read_csv('Mall_Customers.csv') X = df[['Annual Income (k$)', 'Spending Score (1-100)']].values X_scaled = StandardScaler().fit_transform(X) # 2. K selection — ALWAYS run this before fitting wcss, sil = [], [] for k in range(2, 13): km = KMeans(n_clusters=k, init='k-means++', n_init=10, random_state=42) labels = km.fit_predict(X_scaled) wcss.append(km.inertia_) sil.append(silhouette_score(X_scaled, labels)) fig, (ax1, ax2) = plt.subplots(1, 2, figsize=(12, 4)) ax1.plot(range(2, 13), wcss, 'bo-'); ax1.set_title('Elbow Method (WCSS)') ax2.plot(range(2, 13), sil, 'go-'); ax2.set_title('Silhouette Score') # 3. E-Commerce Customer Analytics — RFM Segmentation

Build a production-grade customer segmentation system on real UK e-commerce data. Engineer RFM features from raw transactions, compare K-Means vs GMM vs K-Medoids, and deliver a model that could run in production.

📦 Dataset: UCI Online Retail (541k transactions, 8 columns)  ·  Download here ↗  ·  See all datasets ↑

The core skill: feature engineering

The dataset is raw transactions — you must engineer Recency, Frequency, and Monetary (RFM) features from scratch. This is the key skill: turning a transaction log into a customer-level feature matrix fit for clustering.

RFM is a proven marketing framework used by e-commerce companies worldwide. 📄 Tsai & Chiu (2004) originated formalisation

📚 Reference: Hughes (1994) original RFM paper ↗ — explains why R, F, M each predict repurchase probability independently.

Why compare 3 algorithms?

  • K-Means: Fast, interpretable, centroid-based. Assumes spherical clusters.
  • GMM: Soft assignments (membership probabilities). Handles elliptical clusters. Uses BIC for K selection. 📄 Bishop 2006
  • K-Medoids: Cluster centres are actual data points — more robust to outliers (high-spenders distort K-Means centroids).

What you will produce — RFM feature engineering flow

Common mistake: Students cluster on raw transaction data (541k rows) instead of aggregating first to customer-level RFM (4,339 rows). Always aggregate to your unit of analysis before clustering.
Step 1: Data cleaning (1 hr)
Handle cancelled orders (InvoiceNo starting with 'C'), missing CustomerIDs, negative quantities, UK-only filter. Log how many rows you remove and why — include this in your notebook narrative.
Step 2: RFM feature engineering (1 hr)
Compute snapshot date (max invoice date + 1 day). GroupBy CustomerID: Recency = (snapshot − last_purchase).days, Frequency = count unique invoices, Monetary = sum of (Quantity × UnitPrice). Apply log1p transform to all three (they're heavily right-skewed).
Step 3: Scaling and distribution check (30 min)
StandardScaler on log-transformed RFM. Plot before/after distributions. Verify log transform + scaling gives approximately normal distributions. This is why log transform is essential for RFM data.
Step 4: Multi-algorithm comparison (2 hrs)
Run K-Means (K=4 to 6), GMM (BIC curve for K selection), K-Medoids. Compare: Silhouette, Davies-Bouldin, cluster sizes. Which K and algorithm gives the most interpretable segments?
Step 5: 3D RFM scatter visualisation (30 min)
Plot R vs F vs M in 3D with Plotly. Colour by cluster. This is your headline visual — make it publication-quality. Name the clusters: "Champions", "At Risk", "New Customers", "Hibernating".
Step 6: Business analysis (1 hr)
For each segment: % of customers, % of revenue, RFM profile. Recommend CRM actions: reactivation emails for "At Risk", loyalty rewards for "Champions", onboarding for "New Customers."

Bootstrap stability test

from scipy.optimize import linear_sum_assignment from sklearn.metrics.pairwise import euclidean_distances from sklearn.metrics.cluster import adjusted_rand_score K = 4; n_boot = 30; ari_scores = [] reference_km = KMeans(n_clusters=K, n_init=10, random_state=42).fit(X) for seed in range(n_boot): idx = np.random.RandomState(seed).choice(len(X), size=int(len(X)*0.8), replace=False) km_boot = KMeans(n_clusters=K, n_init=10, random_state=seed).fit(X[idx]) # Hungarian label alignment D = euclidean_distances(km_boot.cluster_centers_, reference_km.cluster_centers_) _, col_ind = linear_sum_assignment(D) mapping = {new:old for new, old in enumerate(col_ind)} remapped = np.array([mapping[l] for l in km_boot.predict(X[idx])]) ari_scores.append(adjusted_rand_score(reference_km.predict(X[idx]), remapped)) print(f"Bootstrap ARI: {np.mean(ari_scores):.3f} ± {np.std(ari_scores):.3f}") print(f"Stability: {'STABLE ✓' if np.mean(ari_scores) > 0.7 else 'UNSTABLE ✗ — try fewer K'}")
Expected results for Project 2 (RFM): Silhouette 0.40–0.60 · K=4 standard RFM segments · Champions typically 5–15% · At Risk 15–25% · Loyal 25–35% · New/Occasional 30–45%. Log-transform is non-negotiable — without it expect one giant cluster. Bootstrap ARI should be >0.75 for K=4.

  • Forgetting to use log1p (handles zeros) — customers with only 1 purchase get log(0) = -inf
  • Including cancelled transactions (InvoiceNo starts with 'C') in Frequency and Monetary
  • Not removing customers with negative total spend (net returns) — they form a spurious cluster

Intermediate DBSCAN HDBSCAN Anomaly Detection
Credit Card Fraud Anomaly Detection

Use density-based clustering to detect fraudulent transactions. DBSCAN and HDBSCAN find outliers — points that don't belong to any dense cluster — without needing fraud labels.

📦 Dataset: Kaggle Credit Card Fraud (284,807 transactions, 0.17% fraud)  ·  Download here ↗  ·  See all datasets ↑

Why density-based for fraud?

K-Means assigns every point to a cluster. DBSCAN identifies points that don't fit any dense neighbourhood as noise (label = -1) — these are your fraud candidates. The algorithm requires no fraud labels, making it practical when labels are unavailable or expensive.

📚 Ester et al. (1996) original DBSCAN paper — KDD Test of Time award. ACM ↗

DBSCAN vs HDBSCAN for this task

  • DBSCAN: Single epsilon — requires careful tuning. Uniform density assumption.
  • HDBSCAN: Variable density clusters. More robust to transactions that cluster at different densities across time periods. 📄 McInnes 2017
  • Compare noise rates: if HDBSCAN noise rate ≈ true fraud rate (0.17%), it is detecting the right outliers.

DBSCAN concept — core and border points illustrated

The key insight: Fraud transactions are rare and isolated — they don't form dense neighbourhoods. DBSCAN labels them as noise (−1). Your evaluation: compare DBSCAN noise labels against the Class column (0=normal, 1=fraud). Precision and recall tell you how well noise correlates with actual fraud.
Step 1: Subsample and EDA (30 min)
Full 284k rows is slow for DBSCAN. Stratified sample: 5k normal + all 492 fraud. Check class imbalance. Plot Time and Amount distributions split by Class.
Step 2: Feature scaling (15 min)
V1–V28 are already PCA-transformed. Scale Time and Amount with RobustScaler (outlier-robust for fraud data).
Step 3: DBSCAN epsilon tuning (1 hr)
Use k-distance graph (k=minPts, plot sorted k-nearest-neighbour distances). The "knee" in this plot is your epsilon. Run DBSCAN for 5 epsilon values around the knee. Track: noise fraction vs. known fraud rate (0.17%).
Step 4: HDBSCAN comparison (1 hr)
Run HDBSCAN with min_cluster_size=5. Compare noise labels with DBSCAN. Compute: precision (noise∩fraud / noise), recall (noise∩fraud / all_fraud) for both algorithms.
Step 5: Isolation Forest baseline (30 min)
Fit IsolationForest(contamination=0.002) as a supervised-style baseline. Compare AUC-ROC: DBSCAN vs HDBSCAN vs Isolation Forest. Which detects fraud best unsupervised?
Step 6: UMAP visualisation (30 min)
UMAP(n_components=2, n_neighbors=30) for 2D view. Plot with colour = DBSCAN label. Overlay actual fraud labels to show alignment. 📄 UMAP docs

Most "fraud" in this dataset is in the noise points — but 70–80% of noise points are NOT fraud.
Context on class imbalance: The credit card fraud dataset is 99.83% legitimate / 0.17% fraud. This is intentional — unsupervised methods should not know about labels during training. This is the real-world constraint unsupervised fraud detection works under.
Common errors in P3:
  • Including the 'Time' and 'Amount' raw features without scaling — Amount has extreme outliers, Time is cyclical
  • Using StandardScaler instead of RobustScaler — fraud transactions have extreme values that distort means
  • Treating all noise points as fraud — noise fraction of 1% on 284k rows = 2,840 noise points, but only 492 are actual fraud
  • Evaluating with accuracy — a model predicting 100% legitimate gets 99.83% accuracy. Use precision, recall, AUPRC

Build a state-of-the-art NLP topic clustering system: encode 120k articles with sentence-BERT, reduce to 5D with UMAP, cluster with HDBSCAN, and compare against BERTopic. The 2025 production NLP approach.

Why the BERT → UMAP → HDBSCAN pipeline?

Raw text → TF-IDF → K-Means worked in 2015. Today's standard is: sentence transformer embeddings (768D semantic space) → UMAP dimensionality reduction → HDBSCAN. This approach captures semantic meaning, not just word frequency.

📚 Reimers & Gurevych (2019) Sentence-BERT paper — the embedding model this project uses. arXiv ↗

BERTopic — the production library

BERTopic wraps the BERT → UMAP → HDBSCAN pipeline with automatic topic labelling (c-TF-IDF), topic merging, and a clean API. Compare your manual pipeline against BERTopic's automated version.

📚 Grootendorst (2022) BERTopic paper. arXiv ↗ | Docs ↗

The NLP clustering pipeline — step by step

Critical: Never run HDBSCAN directly on 768D embeddings. The curse of dimensionality makes all distances nearly equal in 768D. UMAP first reduces to 5D preserving local semantic structure. 📄 McInnes et al. 2018 UMAP paper
Step 1: Setup and embedding generation (2 hrs)
Install sentence-transformers. Load AG News. Generate embeddings with SentenceTransformer('all-MiniLM-L6-v2'). Save embeddings to disk — this is your most time-consuming step. Use batch_size=64, GPU if available.
Step 2: UMAP dimensionality reduction (30 min)
Reduce 768D → 5D with UMAP(n_components=5, metric='cosine', n_neighbors=15). Fit on training data only. Save the fitted UMAP transform — you will need it to process new articles without refitting.
Step 3: HDBSCAN clustering (30 min)
Cluster 5D UMAP output with HDBSCAN(min_cluster_size=50). Check: how many topics found? What fraction labelled noise? Expected: 8–15 coherent topics, 10–20% noise.
Step 4: Topic labelling (1 hr)
For each cluster, compute c-TF-IDF (class-based TF-IDF): extract top 10 keywords per cluster from article texts. Name each topic based on keywords. Validate against AG News ground truth using NMI.
Step 5: BERTopic comparison (1 hr)
Run BERTopic() on same data. Compare: number of topics found, NMI, top keywords, noise fraction. BERTopic essentially automates your Step 2–4 pipeline with additional topic merging.
Step 6: UMAP 2D visualisation (30 min)
Separately fit UMAP(n_components=2) for visualisation. Plot 2D scatter coloured by cluster. This is your showcase visual — shows semantic clusters in 2D space.

Common errors in P4:
  • Clustering raw BERT embeddings (768D) without UMAP — curse of dimensionality, all silhouettes near 0
  • Using UMAP n_components=2 for clustering (vs 2 for visualisation) — 2D loses too much structure
  • Using metric='euclidean' in UMAP — use metric='cosine' for text embeddings, then 'euclidean' in HDBSCAN on UMAP output
  • Setting min_cluster_size too small (<20) — creates hundreds of micro-clusters
  • Not caching embeddings — sentence-BERT encoding 20k articles takes 3–8 minutes; without caching you repeat this every run

MLflow experiment tracking — complete pattern

import mlflow, mlflow.sklearn mlflow.set_tracking_uri("http://localhost:5000") mlflow.set_experiment("nyc-taxi-zone-clustering") with mlflow.start_run(run_name=f"eps{eps}_min{min_s}"): db = DBSCAN(eps=eps/EARTH_RADIUS_KM, min_samples=min_s, algorithm='ball_tree', metric='haversine') labels = db.fit_predict(X_rad) n_clusters = labels.max() + 1 noise_frac = (labels == -1).mean() sil = silhouette_score(X_scaled, labels, sample_size=10000) mlflow.log_params({"eps_km": eps, "min_samples": min_s}) mlflow.log_metrics({"n_clusters": n_clusters, "noise_frac": noise_frac, "silhouette": sil}) mlflow.sklearn.log_model(db, "dbscan_model") print(f"eps={eps}km, min_s={min_s} → {n_clusters} zones, noise={noise_frac:.1%}, sil={sil:.3f}") # FastAPI serving endpoint — load once at startup from fastapi import FastAPI from pydantic import BaseModel import joblib, numpy as np app = FastAPI() bundle = joblib.load("taxi_zone_model.pkl") # load ONCE at startup class Coord(BaseModel): lat: float; lon: float @app.post("/v1/zone") def get_zone(c: Coord): # Find nearest cluster centroid (DBSCAN has no predict) point = np.radians([[c.lat, c.lon]]) dists = np.linalg.norm(bundle["centroids"] - point, axis=1) zone_id = int(np.argmin(dists)) zone_name = bundle["zone_names"].get(zone_id, f"Zone {zone_id}") dist_km = float(np.min(dists) * 6371) return {"zone_id": zone_id, "zone_name": zone_name, "dist_km": round(dist_km, 2)}
