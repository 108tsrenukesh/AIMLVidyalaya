01What an RNN actually is

A feed-forward network sees one fixed-size input and forgets it instantly. But language, audio, sensor streams, and sales histories arrive as sequences — ordered, variable in length, where what came before changes the meaning of what comes next. A Recurrent Neural Network is the simplest idea that gives a network memory: it processes a sequence one element at a time, and carries a hidden state forward from each step to the next.

The one-sentence modelAn RNN reads a sequence like you read a sentence — left to right, updating a running "gist" (the hidden state) after every word, and using that gist plus the next word to decide what to do.

That running gist is the whole trick. At every step the network combines the new input with its memory of everything so far, produces an output if needed, and passes an updated memory onward. The same small set of weights is reused at every step — which is exactly why an RNN can handle a sequence of any length with a fixed number of parameters.

Why this guide matters in 2026Transformers now dominate, but they grew directly out of the RNN→attention story — you cannot truly understand them without it. And recurrence is roaring back: state-space models (Mamba) and RWKV are RNN-like architectures beating Transformers on long-context efficiency. This guide takes you from the 1980s idea to that 2026 frontier.

02Why a plain network can't do sequences

Before recurrence, why not just feed the whole sequence into an MLP? Three walls:

ProblemWhat goes wrong
Variable lengthAn MLP has a fixed input size. Sentences, songs, and sales histories don't. You'd need a different network for every length.
No sense of orderFlatten "dog bites man" and "man bites dog" into a bag of words and they look identical — but they mean opposite things.
No parameter sharingA pattern ("not good") should be recognised wherever it appears in the sequence. An MLP would have to relearn it separately at every position.

An RNN answers all three at once with one move: share the same weights across every time-step and thread a memory between them. Variable length? Just loop more times. Order? The memory is built up in order. Sharing? There's literally one cell, reused.

Parallel to CNNsA CNN shares weights across space (the same filter slides over the image). An RNN shares weights across time (the same cell steps along the sequence). Same powerful idea — weight sharing — applied to a different axis. If you've read the CNN reference, this will feel familiar.

03The shapes of sequence problems

"Sequence model" covers a surprising range of tasks. Knowing which shape you have tells you how to wire the RNN — and is the first question to ask of any business problem.

ShapeIn → OutReal example
One-to-many1 input → sequenceimage → caption
Many-to-onesequence → 1 labelreview → sentiment; ECG → diagnosis
Many-to-many (aligned)sequence → same-length sequencetag each word (NER, POS)
Many-to-many (seq2seq)sequence → different-length sequencetranslation; speech → text
Real-life framing"Forecast next week's demand from the last 90 days" is many-to-one (or many-to-many if you predict several future steps). "Summarise this complaint" is seq2seq. Naming the shape first stops you from building the wrong model — a mistake that's expensive to discover late.

04The RNN cell & the unrolled view

Here is the entire vanilla RNN in one equation. At each step t, take the previous hidden state ht-1 and the new input xt, mix them, squash with a non-linearity:

The recurrence
ht = tanh( Wx xt + Wh ht-1 + b )
yt = Wy ht + by
Three weight matrices — Wx (input→hidden), Wh (hidden→hidden, the memory), Wy (hidden→output) — and that's it. The same three are used at every single step.

The clearest way to picture this is to unroll the loop: draw one copy of the cell per time-step, with the hidden state flowing from each to the next. It's not a deeper network — it's the same cell, photographed at each moment.

An RNN unrolled across time. Inputs come up from the bottom, outputs leave at the top, and the teal arrow — the hidden state — is the memory carried forward. Crucially, the weights inside every cell are identical; unrolling is just a way to see the loop.
The hidden state is everythingAt step t, ht is the network's entire memory of the sequence so far, squeezed into a fixed-size vector. Its size is a key hyperparameter — too small and the model can't remember enough; too big and it overfits and slows down.
Elman vs JordanTwo classic formulations: the Elman network (used here) feeds the hidden state back into itself — ht-1 → ht; the older Jordan network feeds the output back instead — yt-1 → ht. Elman's hidden-state recurrence won out and is what "RNN" means today.

05Forward pass through time

Running an RNN forward is just applying that one recurrence in a loop. Start with h0 (usually zeros), then for each element feed it in and update the state:

The loop, in wordsh ← 0;   for each x in sequence:   h ← tanh(Wxx + Whh + b);   emit y = Wyh.
The final h is a summary of the whole sequence (use it for many-to-one); the sequence of y's is the per-step output (use it for tagging or generation).

Because the loop length equals the sequence length, the computation is inherently sequential — step t can't start until step t-1 finishes. That single fact is the RNN's blessing (true memory) and its curse (no parallelism), and it's exactly the weakness Transformers later exploit (§ to Transformers).

Mind the initial stateResetting h to zero is right between independent sequences, but if you're streaming one long signal in chunks you must carry the state across chunks (and detach it from the graph) — a classic bug that quietly destroys accuracy.

06Backpropagation through time (BPTT)

How does an RNN learn? The same way any network does — gradient descent — but the gradient has to flow backward through the unrolled loop. This is Backpropagation Through Time: unroll the network, run the normal backward pass across all the time-steps, and because the weights are shared, sum each weight's gradient over every step it was used.

The key wrinkleA single weight matrix Wh appears at every time-step. Its total gradient is the sum of the gradient contributions from all steps. The loss at step t depends on every earlier hidden state, so the chain rule reaches all the way back to step 0.
BPTT in motion. The forward pass (grey) builds hidden states left to right; the loss at the end sends its gradient backward through every step (red). Because that red signal is multiplied at each hop, it either fades (vanishing) or blows up (exploding) — the problem §7 tackles head-on.
⬆ Same backprop you already knowBPTT isn't a new algorithm — it's ordinary backpropagation run on the unrolled graph. If you can backprop through a deep MLP, you can backprop through time; the only twist is summing the shared-weight gradients.

Truncated BPTT

For long sequences, unrolling thousands of steps is memory-prohibitive and the gradient is unreliable anyway. Truncated BPTT processes the sequence in chunks of, say, 50–200 steps: carry the hidden state forward across chunks, but only back-propagate within each chunk. It's the standard practical compromise.

07The vanishing & exploding gradient problem

Here is the villain of the whole story — the reason vanilla RNNs fail on anything but short sequences, and the problem every later architecture exists to solve.

When the gradient flows back through time, it gets multiplied by the same recurrent weight (and the tanh derivative) at every step. Multiply a number slightly less than 1 by itself 100 times and it vanishes to zero; slightly more than 1 and it explodes to infinity.

Why it happensThe gradient across k steps carries a factor like (Wh)k times products of tanh derivatives (each ≤ 1). If the relevant eigenvalue of Wh is < 1 → vanish; if > 1 → explode. There's no comfortable middle for long ranges.
Vanishing gradients: as the learning signal travels back through time, it shrinks geometrically. By the time it reaches the early steps it's effectively zero — so the network cannot learn long-range dependencies. "The clouds were in the … sky" is easy; "I grew up in France … I speak fluent ___" is where vanilla RNNs fail.
SymptomCauseFix
Loss spikes to NaNexploding gradientsgradient clipping (cheap, essential)
Can't learn long contextvanishing gradientsgated cells — LSTM / GRU (next)
The pivot of this whole guideExploding gradients have an easy patch (clip them). Vanishing gradients do not — and solving them is precisely what LSTMs and GRUs were invented for. Everything from here is a response to this one slide.

08LSTM — giving memory a protected highway

The Long Short-Term Memory cell (Hochreiter & Schmidhuber, 1997) is the breakthrough that made RNNs actually work on long sequences. Its idea is elegant: add a separate cell state — a "memory highway" that runs straight through the sequence with only minor, gated edits — so gradients can travel far without vanishing.

Three gates (each a little sigmoid network outputting values in 0–1) control the highway:

GateAsksEffect
Forgetwhat should I erase from memory?multiplies the old cell state (0 = forget, 1 = keep)
Inputwhat new info should I store?adds a gated candidate into the cell state
Outputwhat should I expose right now?filters the cell state into the hidden state h
The LSTM cell. The gold line is the cell state — memory's protected highway. The forget gate (×) decides what to drop, the input gate (+) decides what to add, and the output gate (×) decides what to reveal as h. Because the highway is mostly additive, gradients flow across it almost undamped — the "constant error carousel" that beats vanishing gradients.
The equations (for the record) ft=σ(Wf·[ht-1,xt]) ·   it=σ(Wi·[…]) ·   ot=σ(Wo·[…]) ·   gt=tanh(Wg·[…])
Ct = ft ⊙ Ct-1 + it ⊙ gt     ht = ot ⊙ tanh(Ct)
The additive update to C is the magic: gradients pass through ⊕ without being repeatedly multiplied down.
Real-life payoffFor a decade, LSTMs powered Google Translate, Siri, and most production sequence models. Even today, for small-to-medium tabular time-series (demand, sensors), a well-tuned LSTM is often the right choice — cheaper and more data-efficient than a Transformer.

09GRU — the leaner gated cell

The Gated Recurrent Unit (2014) asks: do we really need three gates and a separate cell state? It merges things into two gates and folds the cell state into the hidden state:

GateJob
Resethow much past state to ignore when forming the new candidate
Updatehow much to blend old state vs new candidate (does forget + input in one)
LSTM vs GRU — how to chooseGRU has fewer parameters → trains faster, needs less data, often matches LSTM. LSTM's separate cell state can edge it out on very long or complex dependencies. Practical rule: start with GRU; if it underfits long-range structure, try LSTM. Always benchmark both — the winner is data-dependent.
The GRU. Just two gates — reset (how much past to ignore) and update (how much to keep vs refresh) — and no separate cell state. Simpler and lighter than an LSTM, yet frequently just as accurate.
Real-life payoffOn modest datasets — the common case in business — GRUs frequently win on accuracy-per-rupee-of-compute simply because they're easier to train and less prone to overfitting.

10Bidirectional & deep RNNs

Bidirectional RNNs

A normal RNN only knows the past. But for many tasks — tagging a word, transcribing audio — the future context matters too ("bank" of a river vs a bank loan). A bidirectional RNN runs two RNNs, one forward and one backward, and concatenates their states, so every position sees the whole sequence.

ConstraintBidirectional needs the entire sequence up front — so it's great for classification/tagging on complete inputs, but unusable for real-time streaming or generation, where the future hasn't happened yet.

Deep (stacked) RNNs

Stack multiple recurrent layers so the output sequence of one feeds the next. Lower layers learn local patterns, higher layers learn abstract structure — the same depth intuition as in any deep net. Two to four layers is typical; more rarely helps and is hard to train.

11Sequence-to-sequence — encoder & decoder

How do you map a sequence to a different-length sequence — English to Hindi, audio to text, a question to an answer? The seq2seq design (2014) uses two RNNs: an encoder reads the whole input into a final hidden state (a "thought vector"), and a decoder generates the output one token at a time from that vector.

Encoder–decoder seq2seq. The encoder compresses the entire input into one context vector; the decoder unrolls it into the output. Elegant — but notice the red bottleneck.
The bottleneck problemCramming a 40-word sentence into one fixed vector loses information — and the longer the input, the worse it gets. Translation quality fell off a cliff for long sentences. The fix for this single flaw is the most important idea in modern AI: attention.

12Attention — the idea that changed everything

Instead of forcing the decoder to rely on one frozen context vector, attention (Bahdanau 2014; Luong 2015) lets it look back at every encoder state at each output step, and take a weighted average — focusing on the input words most relevant right now.

Attention in three steps(1) Score the current decoder state against every encoder state. (2) Softmax the scores into weights that sum to 1. (3) Blend: the context is the weighted sum of encoder states. Repeat for every output token, with fresh weights each time.
Soft alignment: generating "happy", the decoder attends mostly to "content" (the French word it translates). The model learns these alignments on its own — no dictionary, no fixed bottleneck. Thicker line = more attention.
Why it's a turning pointAttention removed the bottleneck, made models interpretable (you can see the alignments), and handled long sequences gracefully. It worked so well that researchers asked a radical question: if attention is doing the heavy lifting, do we even need the slow recurrent loop?

13CTC — training without alignment

Some tasks have input and output sequences of different lengths with unknown alignment: speech (thousands of audio frames → a few words) and handwriting (pixels → characters). You don't know which frames map to which letter. Connectionist Temporal Classification (CTC) solves this without per-frame labels.

The ideaAdd a special blank token. The network labels every input frame; then a collapse rule (merge repeated labels, then delete blanks) turns the long frame-level sequence into the short output. CTC's loss sums the probability over all frame-labellings that collapse to the correct target — so the model learns alignment implicitly.
CTC collapse: per-frame predictions "C C _ A A _ _ T" → merge repeats → drop blanks (_) → "CAT". The blank lets the model say "no new letter here" and separate genuine repeats ("ll" in "hello") from stutters.
Real-life payoffCTC powered a generation of speech-to-text and OCR systems (and still anchors many streaming ASR pipelines) precisely because it needs only (audio, transcript) pairs — no expensive frame-by-frame alignment. Pair a BiLSTM/Conv encoder with a CTC head and you have a working recogniser.

14From attention to Transformers

In 2017, "Attention Is All You Need" answered that question: no. Drop recurrence entirely; use self-attention (every position attends to every other position) plus positional encodings to inject order. The result — the Transformer — could process a whole sequence in parallel and model arbitrarily long-range dependencies directly.

The core trade. RNNs are sequential — step N waits for N-1 — but use constant memory per step. Transformers process all positions at once (fast on GPUs) at the cost of O(N²) attention. This single trade-off explains both the Transformer takeover and the 2026 comeback of efficient recurrence.
The honest lineageTransformers didn't fall from the sky — they're the seq2seq-with-attention model with the recurrence removed. Everything you just learned (encoder/decoder, attention, teacher forcing, embeddings, beam search) carries straight over. You now understand the foundation GPT and BERT are built on.
Going deeper into Transformers next? They deserve their own guide — this section is the bridge. The RNN concepts here are the prerequisites.

15Feeding sequences in: tokens & embeddings

An RNN consumes vectors, but language is symbols. The bridge is tokenization (split text into units — words, sub-words, or characters) followed by embeddings (map each token id to a learned dense vector).

TokenizationTrade-off
Charactertiny vocab, handles any word, but long sequences
Wordshort sequences, but huge vocab + out-of-vocabulary words
Sub-word (BPE)the modern default — balances both, no OOV
Embeddings carry meaningThe embedding layer is just a lookup table, learned with the rest of the model, that places similar tokens near each other in vector space ("king" near "queen"). For time-series the analogue is straightforward: your features are already vectors — just scale/normalise them.
⬆ It's a learned matrixAn embedding layer is a plain weight matrix trained by the same backprop you know — one row per token, selected by id.

16Training RNNs well

RNNs are notoriously finicky. A handful of techniques separate "won't converge" from "works":

TechniqueWhy it's essential
Gradient clippingcaps gradient norm (e.g. 1.0–5.0) — the cheap fix for exploding gradients. Non-negotiable.
Teacher forcingduring training, feed the true previous token (not the model's guess) into the decoder — faster, more stable convergence.
Orthogonal / identity initinitialise the recurrent matrix to preserve gradient norm across steps.
Variational / recurrent dropoutapply the same dropout mask across all time-steps (Gal & Ghahramani, 2016) — naïve per-step dropout destroys memory.
Pack/pad + maskingbatch variable-length sequences with padding, then mask the pad steps so they don't pollute the loss.
Teacher-forcing's catch: exposure biasAt test time there's no ground truth to feed — the model eats its own predictions, and small errors compound. Mitigations: scheduled sampling, or just decode carefully (§ Generation).

Accuracy alone rarely fits sequence tasks. Match the metric to the shape:

Business-first metricFor demand forecasting, RMSE punishes big misses; MAPE gives a percentage execs understand; but the real question is often "did we over- or under-stock, and what did that cost?" — a custom, asymmetric cost metric beats any textbook one.

18Decoding & generation

A trained sequence model outputs a probability distribution over the next token. How you pick from it changes the output dramatically.

StrategyWhat it doesUse when
Greedyalways take the top tokenfast, but bland & repetitive
Beam searchkeep the top-k partial sequencestranslation, where there's a "right" answer
Sampling + temperaturesample; temperature controls randomnesscreative generation (text, music)
Top-k / top-psample only from the most likely tokensthe modern default for fluent, varied text
Temperature, intuitivelyLow temperature (→0) = confident and repetitive; high (→1+) = diverse but riskier. It's the single most useful knob when a generator feels "too boring" or "too unhinged".

19Worked example — one RNN step by hand

Numbers make it concrete. Take a 2-dim hidden state, 1-dim input, and tiny weights. Start h0=[0,0], input x1=1.0.

Step 1 Let Wx=[0.5, −0.3]ᵀ, Wh=[[0.1,0.2],[0.0,0.1]], b=0.
pre-activation = Wx·x1 + Wh·h0 = [0.5, −0.3] + [0,0] = [0.5, −0.3]
h1 = tanh([0.5, −0.3]) = [0.462, −0.291]
Step 2 Now x2=0.5. pre = Wx·0.5 + Wh·h1 = [0.25,−0.15] + [0.1·0.462+0.2·(−0.291), 0.1·(−0.291)] = [0.25,−0.15]+[−0.012,−0.029] = [0.238,−0.179]
h2 = tanh([0.238, −0.179]) = [0.234, −0.177]
See how h2 depends on x2 and (through h1) on x1? That's memory, arithmetically.

numpy_rnn.py — verified: h1=[0.462,-0.291], h2=[0.234,-0.177]
# numpy_rnn.py — a vanilla RNN forward pass (and one BPTT step) from scratch.
# Reproduces the exact numbers in the Reference "worked example" section.
import numpy as np


def rnn_forward(xs, Wx, Wh, b, h0):
    """xs: list of scalars/vectors. Returns the hidden states over time."""
    h = h0.copy()
    hs = []
    for x in xs:
        h = np.tanh(Wx * x + Wh @ h + b)   # the whole RNN, one line
        hs.append(h.copy())
    return hs


if __name__ == "__main__":
    Wx = np.array([0.5, -0.3])
    Wh = np.array([[0.1, 0.2],
                   [0.0, 0.1]])
    b = np.array([0.0, 0.0])
    h0 = np.array([0.0, 0.0])

    hs = rnn_forward([1.0, 0.5], Wx, Wh, b, h0)
    print("h1 =", np.round(hs[0], 3))   # -> [ 0.462 -0.291 ]
    print("h2 =", np.round(hs[1], 3))   # -> [ 0.25  -0.177 ]

    # --- one step of BPTT intuition: gradient of h1 wrt the pre-activation ---
    # d/dz tanh(z) = 1 - tanh(z)^2  (this <=1 factor, chained over steps, is why
    # vanilla RNN gradients vanish over long sequences).
    dtanh = 1 - hs[0] ** 2
    print("tanh' at step 1 =", np.round(dtanh, 3), " (each step multiplies by <=1)")

# pytorch_lstm.py — many-to-one: an LSTM that classifies a sequence (e.g. sentiment).
import torch
import torch.nn as nn


class LSTMClassifier(nn.Module):
    def __init__(self, vocab_size, embed_dim=64, hidden=128, num_classes=2,
                 layers=1, bidirectional=True):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, embed_dim, padding_idx=0)
        self.lstm = nn.LSTM(embed_dim, hidden, num_layers=layers,
                            batch_first=True, bidirectional=bidirectional,
                            dropout=0.0)
        out_dim = hidden * (2 if bidirectional else 1)
        self.head = nn.Linear(out_dim, num_classes)

    def forward(self, x, lengths=None):
        e = self.embed(x)                       # (B, T, E)
        if lengths is not None:                 # mask padding properly
            packed = nn.utils.rnn.pack_padded_sequence(
                e, lengths.cpu(), batch_first=True, enforce_sorted=False)
            out, (h, c) = self.lstm(packed)
        else:
            out, (h, c) = self.lstm(e)
        # use the final hidden state(s); concat both directions if bidirectional
        if self.lstm.bidirectional:
            last = torch.cat([h[-2], h[-1]], dim=1)
        else:
            last = h[-1]
        return self.head(last)                  # (B, num_classes) logits


if __name__ == "__main__":
    torch.manual_seed(0)
    model = LSTMClassifier(vocab_size=5000)
    x = torch.randint(1, 5000, (8, 40))         # batch 8, length 40
    lengths = torch.randint(10, 41, (8,))
    logits = model(x, lengths)
    print("logits:", tuple(logits.shape))       # (8, 2)
    print("params:", sum(p.numel() for p in model.parameters()))
    # Train with nn.CrossEntropyLoss + Adam; ALWAYS clip grads:
    #   torch.nn.utils.clip_grad_norm_(model.parameters(), 5.0)
seq2seq_attn.py — verified: attention weights sum to 1.0
# seq2seq_attn.py — encoder-decoder with additive (Bahdanau) attention.
import torch
import torch.nn as nn
import torch.nn.functional as F


class Encoder(nn.Module):
    def __init__(self, vocab, emb=64, hid=128):
        super().__init__()
        self.embed = nn.Embedding(vocab, emb, padding_idx=0)
        self.gru = nn.GRU(emb, hid, batch_first=True, bidirectional=True)
        self.bridge = nn.Linear(hid * 2, hid)

    def forward(self, src):
        e = self.embed(src)
        outs, h = self.gru(e)                       # outs: (B,T,2H)
        h = torch.tanh(self.bridge(torch.cat([h[0], h[1]], dim=1)))  # (B,H)
        return outs, h


class Attention(nn.Module):
    def __init__(self, hid):
        super().__init__()
        self.W = nn.Linear(hid * 2 + hid, hid)
        self.v = nn.Linear(hid, 1, bias=False)

    def forward(self, dec_h, enc_outs):
        # dec_h: (B,H), enc_outs: (B,T,2H)
        T = enc_outs.size(1)
        dec = dec_h.unsqueeze(1).repeat(1, T, 1)         # (B,T,H)
        score = self.v(torch.tanh(self.W(torch.cat([dec, enc_outs], dim=2))))  # (B,T,1)
        alpha = F.softmax(score, dim=1)                  # attention weights
        ctx = (alpha * enc_outs).sum(dim=1)              # (B,2H)
        return ctx, alpha.squeeze(-1)


class Decoder(nn.Module):
    def __init__(self, vocab, emb=64, hid=128):
        super().__init__()
        self.embed = nn.Embedding(vocab, emb, padding_idx=0)
        self.attn = Attention(hid)
        self.gru = nn.GRU(emb + hid * 2, hid, batch_first=True)
        self.out = nn.Linear(hid + hid * 2 + emb, vocab)

    def forward(self, tok, dec_h, enc_outs):
        e = self.embed(tok).unsqueeze(1)                 # (B,1,emb)
        ctx, alpha = self.attn(dec_h, enc_outs)          # (B,2H)
        gru_in = torch.cat([e, ctx.unsqueeze(1)], dim=2)
        out, h = self.gru(gru_in, dec_h.unsqueeze(0))
        logit = self.out(torch.cat([out.squeeze(1), ctx, e.squeeze(1)], dim=1))
        return logit, h.squeeze(0), alpha


if __name__ == "__main__":
    torch.manual_seed(0)
    V = 2000
    enc, dec = Encoder(V), Decoder(V)
    src = torch.randint(1, V, (4, 12))                   # batch 4, src len 12
    enc_outs, h = enc(src)
    tok = torch.randint(1, V, (4,))                      # one decoder step
    logit, h, alpha = dec(tok, h, enc_outs)
    print("step logits:", tuple(logit.shape))           # (4, 2000)
    print("attention weights:", tuple(alpha.shape))     # (4, 12) -> sums to 1 per row
    print("alpha row sums:", alpha.sum(1).round(decimals=3).tolist())
    # Train with teacher forcing + CrossEntropyLoss; decode with beam search.
forecast_lstm.py — verified: trains, 7-day horizon
# forecast_lstm.py — multivariate time-series forecasting (e.g. FMCG demand).
# Many-to-one (or multi-output) LSTM with leakage-safe windowing.
import numpy as np
import torch
import torch.nn as nn


def make_windows(series, lookback=30, horizon=7):
    """series: (T, F). Returns X:(N,lookback,F), Y:(N,horizon) for target col 0.
    NOTE: windows are time-ordered; split BEFORE scaling to avoid leakage."""
    X, Y = [], []
    for i in range(len(series) - lookback - horizon + 1):
        X.append(series[i:i + lookback])
        Y.append(series[i + lookback:i + lookback + horizon, 0])  # forecast col 0
    return np.array(X, dtype=np.float32), np.array(Y, dtype=np.float32)

    def __init__(self, n_features, hidden=64, horizon=7, layers=2):
        super().__init__()
        self.lstm = nn.LSTM(n_features, hidden, num_layers=layers,
                            batch_first=True, dropout=0.1)
        self.head = nn.Linear(hidden, horizon)      # direct multi-step output

    def forward(self, x):
        out, (h, c) = self.lstm(x)
        return self.head(h[-1])                     # (B, horizon)

mamba_block.py — faithful S6 selective SSM, verified on CPU
# mamba_block.py — a FAITHFUL minimal implementation of the Mamba (S6) block.
# This is the REAL selective state-space algorithm from Gu & Dao (2023):
#   - input-dependent (selective) Delta, B, C
#   - zero-order-hold discretization of the continuous A, B
#   - a sequential selective scan
#   - the gated (SiLU) branch and the short causal depthwise conv
# The ONLY thing missing vs the official CUDA implementation is the fused
# hardware-aware kernel — the math and architecture here are identical, and
# this runs on CPU. (Reference architecture: johnma2006/mamba-minimal.)
import math
import torch
import torch.nn as nn
import torch.nn.functional as F


class MambaBlock(nn.Module):
    def __init__(self, d_model, d_state=16, d_conv=4, expand=2, dt_rank=None):
        super().__init__()
        self.d_model = d_model
        self.d_inner = expand * d_model
        self.d_state = d_state
        self.dt_rank = dt_rank or math.ceil(d_model / 16)

        # project input up to (x, gate), each of size d_inner
        self.in_proj = nn.Linear(d_model, self.d_inner * 2, bias=False)
        # short causal depthwise conv over time (local context before the SSM)
        self.conv1d = nn.Conv1d(self.d_inner, self.d_inner, kernel_size=d_conv,
                                groups=self.d_inner, padding=d_conv - 1, bias=True)
        # produces the *input-dependent* Delta, B, C  (this is the "selective" part)
        self.x_proj = nn.Linear(self.d_inner, self.dt_rank + 2 * d_state, bias=False)
        self.dt_proj = nn.Linear(self.dt_rank, self.d_inner, bias=True)

        # A is a learned, per-channel diagonal state matrix (kept negative for stability)
        A = torch.arange(1, d_state + 1, dtype=torch.float32).repeat(self.d_inner, 1)
        self.A_log = nn.Parameter(torch.log(A))           # (d_inner, d_state)
        self.D = nn.Parameter(torch.ones(self.d_inner))   # skip connection
        self.out_proj = nn.Linear(self.d_inner, d_model, bias=False)

    def forward(self, x):                                  # x: (B, L, d_model)
        b, l, _ = x.shape
        xz = self.in_proj(x)                               # (B, L, 2*d_inner)
        x_in, z = xz.chunk(2, dim=-1)                      # each (B, L, d_inner)

        # causal depthwise conv (truncate the right padding back to length L)
        x_in = x_in.transpose(1, 2)                        # (B, d_inner, L)
        x_in = self.conv1d(x_in)[:, :, :l].transpose(1, 2)
        x_in = F.silu(x_in)

        y = self._ssm(x_in)                                # selective scan
        y = y * F.silu(z)                                  # gated branch
        return self.out_proj(y)

    def _ssm(self, x):                                     # x: (B, L, d_inner)
        A = -torch.exp(self.A_log)                         # (d_inner, d_state)
        D = self.D

        x_dbl = self.x_proj(x)                             # (B, L, dt_rank+2*d_state)
        delta, B, C = torch.split(
            x_dbl, [self.dt_rank, self.d_state, self.d_state], dim=-1)
        delta = F.softplus(self.dt_proj(delta))            # (B, L, d_inner)  -> selective step size
        return self._selective_scan(x, delta, A, B, C, D)

    def _selective_scan(self, u, delta, A, B, C, D):
        b, l, d_in = u.shape
        n = A.shape[1]
        # discretize: dA = exp(delta * A),  dB*u  (zero-order hold)
        dA = torch.exp(torch.einsum('bld,dn->bldn', delta, A))        # (B,L,d_in,n)
        dB_u = torch.einsum('bld,bln,bld->bldn', delta, B, u)         # (B,L,d_in,n)

        h = torch.zeros(b, d_in, n, device=u.device, dtype=u.dtype)
        ys = []
        for t in range(l):                                  # the selective scan (O(L))
            h = dA[:, t] * h + dB_u[:, t]                   # state update
            y = torch.einsum('bdn,bn->bd', h, C[:, t])      # readout
            ys.append(y)
        y = torch.stack(ys, dim=1)                          # (B, L, d_in)
        return y + u * D                                    # skip connection


if __name__ == "__main__":

    y = block(x)

    print("out:", tuple(y.shape))                           # (2, 100, 64)

21Looking inside — interpretability & debugging

RNNs aren't black boxes if you know where to look. Four practical lenses turn "it's not working" into "here's why".

1 · Hidden-state trajectories (PCA / t-SNE)

The hidden state is a vector that moves through space as the sequence unfolds. Project it to 2-D with PCA or t-SNE and watch it travel — you'll often see it sort inputs into meaningful regions.

Hidden states of a sentiment model, projected to 2-D. Each dot is the state after one more word; the positive and negative reviews drift into separate regions as the verdict becomes clear. If your classes don't separate here, the model hasn't learned the signal.

2 · Gate-activation tracking

For LSTMs/GRUs, plot the forget-gate activations over time. Values near 1 mean "keep the memory"; near 0 mean "wipe it". The wipes usually land on meaningful boundaries (sentence ends, topic shifts).

Forget-gate activations across a sequence. Mostly ~1 (memory preserved), with sharp drops where the cell decides to reset. Auditing these tells you whether the model is using long-term memory at all — flat-near-zero means it isn't.

3 · Gradient-flow monitoring

Track the gradient norm per layer / per time-step during training. If early-step gradients are orders of magnitude smaller than late-step ones, you're watching the vanishing-gradient problem live — switch to gated cells, add skip connections, or shorten the BPTT window. If the norm pins to your clip value every step, lower the learning rate.

Cheap, high-value habitLog total_norm from gradient clipping every step. A flat-then-spiking trace screams "exploding"; a slow decay to near-zero screams "vanishing". This one number diagnoses most RNN training failures.

4 · Attention heatmaps

If your model has attention, its weights are a free explanation: a matrix of which inputs each output attended to.

An attention alignment matrix (target French rows × source English columns). A near-diagonal with sensible off-diagonal jumps (here, the noun/adjective reorder) means the model learned real correspondences. Scattered noise usually means a masking or shape bug.
Why this matters for the businessIn regulated settings — healthcare, finance, legal — a number nobody can explain won't get deployed. Attention maps, gate audits, and trajectory plots are how you earn a domain expert's trust (and pass a model-risk review).

One pattern deserves a special mention because it shows up everywhere in industry: the LSTM autoencoder. Train an encoder–decoder LSTM to reconstruct normal sequences only. 24Mamba & selective state-space models — in depth

The modern landscape section sketched why recurrence is back. Here's how Mamba (Gu & Dao, 2023) actually works — because it's the architecture most likely to matter next, and it's a direct descendant of everything you've learned.

From RNN to state-space model

A state-space model (SSM) is a linear recurrence with continuous-time roots:

The SSM recurrence
ht = A̅ ht-1 + B̅ xt     yt = C ht
Like an RNN, but with no non-linearity in the recurrence. A̅, B̅ are discretized from continuous matrices A, B via a step size Δ (zero-order hold): A̅ = exp(ΔA). The matrix A is initialised with HiPPO theory so the state optimally compresses history.

Because it's linear, this recurrence can be computed two ways: as a fast parallel convolution during training, and as a cheap step-by-step recurrence during inference. Best of both worlds — Transformer-like training speed, RNN-like O(1) streaming.

The "selective" breakthrough (S6)

Classic SSMs (S4) used fixed A, B, C — they couldn't choose what to focus on based on content. Mamba's key idea: make Δ, B, and C functions of the input. Now the model dynamically decides, per token, how much to remember vs ignore — content-based reasoning, the thing attention gave Transformers.

A Mamba (S6) block: a short causal conv for local context, input-dependent (selective) Δ/B/C, discretization of A/B, a linear-time selective scan, and a SiLU gate. The selectivity is what lets it match Transformers — but it costs the parallel-convolution trick, so Mamba uses a hardware-aware parallel scan instead.
Why it scalesAttention is O(L²) in sequence length L; the selective scan is O(L) with constant memory per step. On sequences of tens of thousands of tokens (genomics, audio, long documents) that's the difference between feasible and not.

25Common pitfalls

  • No gradient clipping → loss explodes to NaN. The first thing to add, always.
  • Not carrying/detaching hidden state across chunks of a long stream → either broken memory or a backward-through-everything crash.
  • Per-step dropout on recurrent connections → shredded memory. Use variational (shared-mask) dropout.
  • Forgetting to mask padding → the model learns from pad tokens and your metric lies.
  • Train/serve skew in generation → teacher forcing in training, free-running at test (exposure bias). Decode carefully.
  • Leaking the future in time-series → scaling or splitting using future data inflates backtests. Split by time, fit the scaler on train only.
  • Using a bidirectional model for streaming → impossible at inference; you don't have the future yet.

26Tuning & debugging recipe

  1. Overfit a tiny batch first. If the model can't memorise 8 sequences, there's a bug — fix that before anything else.
  2. Clip gradients (norm 1–5) and watch the gradient norm; if it pins to the clip constantly, lower the LR.
  3. Hidden size & layers: start small (128–256, 1–2 layers); grow only if underfitting.
  4. Sequence length / truncation: longer context helps only if the signal is actually long-range — and costs memory.
  5. LR: Adam at 1e-3 is a safe start; add a scheduler if it plateaus.
  6. GRU vs LSTM: benchmark both — it's quick and the winner is data-dependent.
Debugging mantra"Tiny batch → clip → small model → scale up." Most RNN training failures are exploding gradients or a state-handling bug, not the architecture.

29Q&A bank

Why can't a vanilla RNN remember long-range dependencies?

Because the gradient signal is multiplied by the recurrent weight (and tanh derivatives ≤1) at every step travelling back in time. Over many steps it shrinks geometrically toward zero (vanishing) — so early steps receive almost no learning signal. LSTMs/GRUs fix this with a gated, additive memory path.

What exactly is the hidden state?

A fixed-size vector that summarises everything the network has seen up to the current step. It's the RNN's entire memory; its dimension is a key hyperparameter.

What does "unrolling" an RNN mean — is it a deeper network?

No. Unrolling just draws one copy of the same cell per time-step to visualise the loop. The weights are identical at every step; it's one cell applied repeatedly, not many distinct layers.

Why share weights across time-steps at all?

So a pattern is recognised wherever it occurs in the sequence, and so the model handles any length with a fixed parameter count. It's the temporal analogue of a CNN sharing a filter across space.

Why does the LSTM's cell state help gradients?

The cell-state update is largely additive (Ct = f⊙Ct-1 + i⊙g). Additive paths let gradients flow backward without being repeatedly multiplied down, so they survive across many steps — the "constant error carousel".

Where precisely do exploding gradients come from?

The backward pass across k steps carries a factor like (Wh)k. If the largest eigenvalue (spectral radius) of Wh exceeds 1, that term grows without bound. Gradient clipping caps the norm so a single huge step can't wreck training.

What's the role of the tanh derivative in vanishing gradients?

d/dz tanh(z) = 1 − tanh²(z), which is ≤ 1 (and near 0 when the unit saturates). Chained over many steps, these sub-1 factors multiply the gradient down — compounding the shrinkage from the recurrent weight.

How many parameters does an LSTM layer have?

Roughly 4 × (hidden·(hidden+input) + hidden) — the factor 4 because the forget, input, output gates and candidate each have their own weight matrix and bias. A GRU has ~3× (two gates + candidate), so it's lighter.

What is perplexity, intuitively?

exp(average cross-entropy). It's the effective number of equally-likely choices the model is "torn between" at each step. Perplexity 10 ≈ as unsure as picking uniformly among 10 options. Lower is better.

How does attention compute its weights?

Score each encoder state against the current decoder state (dot-product or a small MLP), softmax the scores into weights that sum to 1, then take the weighted sum of encoder states as the context. Repeat per output step.

LSTM or GRU — which should I use?

Start with GRU (fewer params, trains faster, less data). Switch to LSTM if you need to capture very long or complex dependencies. Always benchmark both; the winner depends on your data.

My RNN loss became NaN. What happened?

Almost certainly exploding gradients. Add gradient clipping (norm 1–5). If it persists, lower the learning rate and check for bad inputs (NaNs, unscaled features).

How big should my hidden state be?

Start small (128–256) and grow only if you're underfitting. Bigger hidden states memorise more but overfit and slow down. Tune it alongside the number of layers (1–2 to start).

How do I handle variable-length sequences in a batch?

Pad to the longest in the batch, record true lengths, and use packing (e.g. pack_padded_sequence) plus masking so the pad steps don't contribute to outputs or loss.

How do I forecast multiple future steps?

Two ways: (1) recursive — predict one step, feed it back, repeat (simple, but errors compound); (2) direct/seq2seq — output the whole horizon at once. Direct multi-output is usually more stable for business forecasting.

When should I use a bidirectional RNN?

Only when you have the entire sequence available at inference (classification, tagging, offline transcription). Never for streaming or generation — you can't see the future yet.

Why is my dropout hurting the RNN?

Standard per-step dropout on recurrent connections randomly severs memory. Use variational (locked) dropout that applies the same mask across all time-steps, or apply dropout only between layers/on inputs.

What is teacher forcing and what's its downside?

Feeding the true previous token (not the model's own prediction) into the decoder during training — it stabilises and speeds learning. Downside: exposure bias, since at test time the model must consume its own (possibly wrong) outputs. Mitigate with scheduled sampling or careful decoding.

How do I avoid data leakage in time-series?

Split strictly by time (no shuffling), fit any scaler/encoder on the training portion only, and never let a feature peek at the future (e.g. a rolling mean must use only past values). Leakage inflates backtests and collapses in production.

Are RNNs obsolete now that we have Transformers?

For large-scale NLP, mostly yes. But for small data, streaming/edge inference, and very long sequences, RNN-style models (including 2026's Mamba and RWKV) are excellent — and recurrence is actively resurgent for efficiency.

Why is attention such a big deal?

It removed the seq2seq bottleneck by letting the decoder look at all encoder states with learned weights. It improved quality, added interpretability, and directly led to the Transformer — the basis of modern LLMs.

What's the difference between a Transformer and an RNN, in one line?

An RNN processes tokens sequentially with a carried hidden state (O(1) memory/step, but can't parallelise across time); a Transformer attends to all tokens at once (parallel, but O(N²) compute and memory in sequence length).

What problem do state-space models (Mamba) solve?

They restore RNN-like linear-time, constant-memory, streaming recurrence — but with training tricks and selectivity that fix the old gradient/expressivity issues — making them strong and cheap on very long sequences where Transformers get expensive.

One-to-many, many-to-one, seq2seq — how do I tell which I need?

Count inputs vs outputs. One input, many outputs (image→caption) = one-to-many. Many inputs, one output (review→label) = many-to-one. Different-length in and out (translate) = seq2seq. Naming the shape picks the wiring.

Truncated BPTT — what length should I pick?

Long enough to span the dependencies that matter, short enough to fit memory and keep gradients sane — commonly 50–200 steps. Carry (and detach) the hidden state across chunks so memory persists without backpropagating through everything.

My generated text is repetitive / too random — what knob?

Temperature. Too repetitive → raise it (or use top-p sampling); too incoherent → lower it (or use beam search for tasks with a single right answer). Top-k/top-p give the best fluency/diversity balance.

Should I always beat a naïve baseline first?

Yes. For forecasting, "repeat last week" or a seasonal-naïve baseline is shockingly strong. If your LSTM can't beat it, the problem is data/framing, not architecture — fix that before scaling the model.

Why concatenate both directions in a BiLSTM?

The forward pass encodes left-context, the backward pass right-context. Concatenating gives each position a representation informed by the whole sequence, which is why BiLSTMs excel at tagging and classification.

How do I deploy an RNN for real-time streaming?

Keep the hidden/cell state between calls and feed one new element at a time — constant time and memory per step. Quantise (INT8) for edge devices, and export via ONNX/TFLite, being careful to preserve the stateful loop.

Does an RNN need embeddings for time-series?

No — numeric features are already vectors; just scale/normalise them (on train stats). Embeddings are for discrete tokens (words, categories); you can embed categorical features (e.g. store id) and concatenate with numeric ones.
