Project 1 · Beginner · ~3 hrs

Beat the baseline

Before any fancy model, prove you can beat seasonal-naive with classical methods — and, just as important, build the rolling-origin backtest that every later project will reuse. This is the project that teaches you to measure honestly.

What you'll build

A clean, reusable backtesting notebook that fits SeasonalNaive, AutoETS, and AutoARIMA across all 500 store-item series at once with statsforecast, evaluates them on rolling-origin cross-validation, and reports WMAPE, bias, and MASE per model. The deliverable is a leaderboard that answers one question honestly: which models actually beat the naive baseline, and by how much?

Real-world analogue: the first thing any forecasting team should do with a new dataset — establish the floor before investing in complexity. Dataset: Store Item Demand Forecasting ↗. Reinforces: reference Methods (classical) ↗ and Evaluation ↗.

What you'll produce — a backtested model leaderboard
1 · Load & shape to long format (~25 min)
Read the sales CSV; reshape to the three columns every Nixtla tool expects: unique_id (store-item key), ds (date), y (sales). Verify there are no gaps in the daily index per series.
2 · Eyeball a sample (~20 min)
Plot 6 random series. Note trend, weekly + yearly seasonality, and whether any are intermittent (they shouldn't be in this dataset — that's Project 3).
3 · Define the baseline (~15 min)
Add SeasonalNaive(season_length=7) — and conceptually a 365-day one. This is the number to beat; everything else is measured against it.
4 · Fit classical models cross-series (~30 min)
Add AutoETS and AutoARIMA. Run StatsForecast.cross_validation with h=28, 4 windows — that's rolling-origin backtesting in one call.
5 · Score & build the leaderboard (~45 min)
Compute WMAPE, bias, MASE per model over the CV output. Rank models; flag any that fail to beat seasonal-naive. Slice WMAPE by horizon to see degradation.
6 · Write up the finding (~25 min)
One chart (leaderboard) + one paragraph: which model wins, by how much over naive, and where it's weakest (which horizon, which series).

Reshape to long format

import pandas as pd df = pd.read_csv("train.csv", parse_dates=["date"]) df = df.rename(columns={"date":"ds", "sales":"y"}) df["unique_id"] = "s" + df["store"].astype(str) + "_i" + df["item"].astype(str) df = df[["unique_id", "ds", "y"]].sort_values(["unique_id", "ds"])

Fit models + rolling-origin backtest in one call

from statsforecast import StatsForecast from statsforecast.models import SeasonalNaive, AutoETS, AutoARIMA sf = StatsForecast( models=[SeasonalNaive(season_length=7), AutoETS(season_length=7), AutoARIMA(season_length=7)], freq="D", n_jobs=-1, ) # rolling origin: 4 windows of 28-day horizon, stepping by 28 days cv = sf.cross_validation(df=df, h=28, step_size=28, n_windows=4)

Score: WMAPE, bias, MASE leaderboard

import numpy as np models = ["SeasonalNaive", "AutoETS", "AutoARIMA"] naive_mae = np.abs(cv["y"] - cv["SeasonalNaive"]).mean() board = [] for m in models: e = cv[m] - cv["y"] board.append({ "model": m, "WMAPE": np.abs(e).sum() / np.abs(cv["y"]).sum(), "bias": e.mean(), "MASE": np.abs(e).mean() / naive_mae, }) print(pd.DataFrame(board).sort_values("WMAPE"))

Expected results

  • AutoETS and AutoARIMA should both beat SeasonalNaive on WMAPE for this clean, seasonal dataset — typically a 20–35% relative improvement. If they don't, you have a bug (often a frequency or season_length mismatch).
  • MASE for the good models should be well below 1 (≈0.6–0.8); SeasonalNaive's MASE is ≈1 by construction.
  • Bias should be small relative to the mean demand; a large systematic bias signals a seasonality the model isn't capturing.
  • WMAPE should rise with horizon — fold-level WMAPE at h=1–7 lower than h=22–28.

Common errors

Wrong season_length. Daily retail has both weekly (7) and yearly (365) seasonality; using 7 captures the dominant weekly cycle. Setting it to 1 silently disables seasonality and the models lose to naive.
Gaps in the date index. If a series is missing dates, AutoARIMA mis-infers the frequency. Reindex each series to a complete daily range first.
Scoring on a single fold. Tempting, but it's the exact mistake the project exists to cure — always average across the 4 windows and look at the spread.

Deliverables

  • A notebook that goes from raw CSV → long format → cross-validated leaderboard, runnable top to bottom.
  • A single leaderboard chart (WMAPE per model with the naive floor marked) and a 1-paragraph finding.
  • A reusable score(cv_df) function returning WMAPE/bias/MASE — you'll import it in every later project.
Self-check
  • ☐ Does at least one model beat seasonal-naive on the average of 4 folds?
  • ☐ Can you state WMAPE at h=1–7 vs h=22–28?
  • ☐ Is your scoring function leakage-free and reusable?

Project 2 · Intermediate · ~6 hrs

LightGBM at scale — cross-learning pipeline

The centerpiece. Rebuild, in miniature, the approach that swept the M5 competition: one gradient-boosted model trained across all series, fed engineered lag and calendar features, with a Tweedie objective for zero-inflated sales — and a head-to-head of recursive vs direct multi-step forecasting.

What you'll build

A production-shaped forecasting pipeline: feature engineering (lags, rolling stats, calendar, price/promo), a single cross-learned LightGBM with a Tweedie objective, recursive multi-step forecasting, and an honest comparison against the Project 1 baselines on the same rolling-origin folds. Then you'll add a direct (per-horizon) variant and compare error-by-horizon — the exact recursive-vs-direct trade-off the M5 winners exploited.

Real-world analogue: the core demand engine for the dense, promo-driven majority of a retail catalogue. Dataset: M5 Walmart ↗ (start with one store to keep it tractable). Reinforces: reference Methods (ML) ↗ and Pitfalls (leakage) ↗.

The pipeline you'll build
1 · Assemble the panel (~45 min)
Join sales with the calendar (events, SNAP) and price files; melt to long (unique_id, ds, y) plus covariates. Start with one store (~3k series) before scaling out.
2 · Engineer leakage-safe features (~75 min)
Lags (1,7,14,28,364), rolling mean/std on shifted demand, calendar parts, price change, promo/event flags. Every rolling stat preceded by .shift(1) within each series.
3 · Train the global model (~45 min)
One LightGBM across all series with objective="tweedie". Use a time-based train/valid split for early stopping — never random.
4 · Recursive multi-step forecast (~60 min)
Predict day 1, append it, recompute lags, predict day 2 … to h=28. Wrap in a function so the harness can call it.
5 · Backtest vs P1 baselines (~45 min)
Run the same rolling-origin folds as Project 1; add LightGBM to the leaderboard. Confirm it beats AutoETS/ARIMA and naive — and by how much.
6 · Add a direct variant & compare by horizon (~60 min)
Train per-horizon models (or use mlforecast's max_horizon). Plot WMAPE by horizon for recursive vs direct and explain the crossover.

Leakage-safe feature engineering

import pandas as pd, numpy as np def make_features(df): df = df.sort_values(["unique_id", "ds"]).copy() g = df.groupby("unique_id")["y"] for lag in (1, 7, 14, 28, 364): df[f"lag_{lag}"] = g.shift(lag) for w in (7, 28): # shift(1) FIRST = no leakage df[f"rmean_{w}"] = g.shift(1).rolling(w).mean() df[f"rstd_{w}"] = g.shift(1).rolling(w).std() df["dow"] = df["ds"].dt.dayofweek df["week"] = df["ds"].dt.isocalendar().week.astype("int16") df["month"] = df["ds"].dt.month return df

Train one cross-learned model (Tweedie)

import lightgbm as lgb feat = make_features(panel).dropna(subset=["lag_364"]) cols = [c for c in feat.columns if c not in ("y","ds","unique_id")] cutoff = feat["ds"].max() - pd.Timedelta(days=28) tr, va = feat[feat.ds <= cutoff], feat[feat.ds > cutoff] # time split model = lgb.LGBMRegressor(objective="tweedie", tweedie_variance_power=1.1, n_estimators=2000, learning_rate=0.03, num_leaves=128, subsample=0.8, colsample_bytree=0.8) model.fit(tr[cols], tr["y"], eval_set=[(va[cols], va["y"])], callbacks=[lgb.early_stopping(100), lgb.log_evaluation(0)])

Recursive multi-step forecast

def recursive_forecast(model, hist, cols, h=28): hist = hist.copy() preds = [] for step in range(h): nxt = next_day_frame(hist) # one row per series, ds = last+1 feats = make_features(pd.concat([hist, nxt])).tail(len(nxt)) nxt["y"] = model.predict(feats[cols]).clip(min=0) # demand >= 0 preds.append(nxt) hist = pd.concat([hist, nxt]) # feed prediction back as a lag return pd.concat(preds)

Recursive vs direct (mlforecast does the bookkeeping)

from mlforecast import MLForecast from mlforecast.target_transforms import Differences fcst = MLForecast(models={"lgb": lgb.LGBMRegressor(objective="tweedie")}, freq="D", lags=[1,7,14,28,364], lag_transforms={1: [(rolling_mean, 7), (rolling_mean, 28)]}, date_features=["dayofweek","month"]) # recursive by default; max_horizon=H trains direct per-horizon models cv_rec = fcst.cross_validation(panel, h=28, n_windows=4, step_size=28) cv_dir = fcst.cross_validation(panel, h=28, n_windows=4, step_size=28, max_horizon=28)

Expected results

  • The cross-learned LightGBM should beat the classical baselines from Project 1 on WMAPE once price/promo/calendar features are in — often a further 5–15% relative improvement on promo-heavy series.
  • Feature importance should rank recent lags (lag_7, lag_28), rolling means, and price/event flags near the top. If a calendar dummy dominates everything, suspect leakage.
  • Recursive vs direct: recursive is usually competitive at short horizons but degrades faster; direct is steadier at h>14. The averaged combination typically beats either — the M5 finding.
  • On the intermittent tail, the dense-model WMAPE will look worse — that's expected, and exactly why Project 3 exists.

Common errors

Leakage via rolling features. A rolling mean that includes the current day inflates the backtest and dies live. The .shift(1) before .rolling() is non-negotiable. (See reference Pitfalls.)
Random train/valid split. Shuffling time series leaks the future into validation. Always split by date for early stopping.
Forgetting to clip at zero. Demand can't be negative; clip predictions. With a non-Tweedie objective you'll also see more negative predictions.
Trees not extrapolating a trend. If a series trends strongly, add a trend feature or difference the target (the Differences transform), or the forecast will flatten.

Deliverables

  • A pipeline notebook/module: panel assembly → make_features → train → recursive forecast → backtest, reusing Project 1's scoring function.
  • An updated leaderboard with LightGBM added, plus a WMAPE-by-horizon chart for recursive vs direct.
  • A short write-up of feature importance and the recursive/direct trade-off, with the numbers.
Self-check
  • ☐ Does LightGBM beat both AutoETS/ARIMA and seasonal-naive on the same folds?
  • ☐ Can you prove no feature leaks the future?
  • ☐ Can you explain where recursive beats direct and vice versa?

Challenges

Stretch Train quantile models (LightGBM objective="quantile" at 0.5/0.9) and report pinball loss — your first probabilistic forecast.
Hard Reproduce the M5 winner's idea: train models at item-store, store-category, and store-department levels and average them. Does the ensemble beat the single global model?
Hard Scale from one store to all ten with mlforecast and measure training time; this is where global-model throughput becomes the real engineering problem.
Project 3 · Intermediate · ~4 hrs

Intermittent-demand engine

The long tail that Project 2's dense model handles badly. Build a classifier that segments every SKU by ADI/CV² and routes it to the right method — Croston, SBA, or TSB — and demonstrate why TSB is the one that survives obsolescence.

What you'll build

A routing engine: for every SKU, compute ADI and CV², place it in the smooth/intermittent/erratic/lumpy quadrant, and forecast it with the method that quadrant prescribes. You'll run Croston, SBA, and TSB, then construct a deliberately obsolescing item to show — with a plot — that only TSB decays toward zero while Croston/SBA stay frozen.

Real-world analogue: spare-parts and slow-mover planning, and the tail of any FMCG catalogue. Dataset: Online Retail II ↗, aggregated to SKU-week. Reinforces: reference Intermittent & lumpy demand ↗.

Classify → route → forecast
1 · Aggregate to SKU-week (~40 min)
Sum transactional quantities to weekly per StockCode; reindex to a complete weekly grid so zero-demand weeks are explicit zeros, not missing rows. The zeros are the whole point.
2 · Compute ADI & CV² (~35 min)
Per SKU: ADI = periods ÷ non-zero periods; CV² = variance/mean² of the non-zero sizes. Assign the quadrant with the 1.32 / 0.49 cutoffs.
3 · Profile the catalogue (~25 min)
Plot the SKUs on the ADI–CV² plane. You'll typically find most SKUs are intermittent or lumpy — a vivid reminder of why dense models alone are insufficient.
4 · Route & forecast (~50 min)
Run CrostonClassic, CrostonSBA, and TSB via statsforecast on the intermittent/lumpy SKUs; keep ETS for the smooth ones. Backtest with a count-appropriate metric.
5 · The obsolescence demo (~40 min)
Take (or synthesise) an item that sells then dies. Forecast with SBA and TSB; plot both. Show SBA stays flat while TSB decays toward zero.
6 · Evaluate by inventory lens (~30 min)
Beyond point error, simulate a simple order-up-to policy and compare holding + stock-out outcomes — the right way to judge intermittent forecasts.

Classify each SKU by ADI / CV²

import numpy as np def adi_cv2(y): y = np.asarray(y, dtype=float) nz = y[y > 0] if len(nz) < 2: return np.inf, np.inf adi = len(y) / len(nz) cv2 = (nz.std(ddof=1) / nz.mean())**2 return adi, cv2 def quadrant(adi, cv2): if adi < 1.32 and cv2 < 0.49: return "smooth" if adi < 1.32: return "erratic" if cv2 < 0.49: return "intermittent" return "lumpy" seg = (weekly.groupby("unique_id")["y"] .apply(lambda s: quadrant(*adi_cv2(s))))

Route to the right method

from statsforecast import StatsForecast from statsforecast.models import AutoETS, CrostonClassic, CrostonSBA, TSB routes = { "smooth": [AutoETS(season_length=52)], "erratic": [CrostonSBA()], "intermittent": [CrostonClassic(), CrostonSBA()], "lumpy": [CrostonSBA(), TSB(alpha_d=0.2, alpha_p=0.2)], } out = {} for q, models in routes.items(): ids = seg[seg == q].index sub = weekly[weekly.unique_id.isin(ids)] if len(sub): out[q] = StatsForecast(models=models, freq="W", n_jobs=-1).forecast(df=sub, h=8)

Obsolescence: TSB decays, Croston/SBA don't

# item sells for 40 weeks, then goes dead for 20 y = np.r_[np.random.poisson(3, 40), np.zeros(20)] dead = pd.DataFrame({"unique_id":"DEAD", "ds": pd.date_range("2025-01-05", periods=60, freq="W"), "y": y}) sf = StatsForecast(models=[CrostonSBA(), TSB(0.2,0.2)], freq="W") fc = sf.forecast(df=dead, h=8) # CrostonSBA ~ frozen at its last positive-demand rate; # TSB has decayed toward 0 because probability was smoothed down each zero week. print(fc)

Expected results

  • Most real-catalogue SKUs land in the intermittent or lumpy quadrants — often well over half. This is the headline insight to show stakeholders.
  • On the obsolescence demo, TSB's forecast is markedly lower (decaying toward zero) than CrostonSBA's, which stays near the last active rate. That gap is the whole reason TSB exists.
  • SBA's forecasts should be slightly lower than Croston's on the same series (the (1−α/2) de-biasing).
  • Using MAPE here will produce infinities — confirm your metric is WMAPE/MASE or an inventory measure, not MAPE.

Common errors

Dropping zero weeks. If you aggregate and only keep weeks with sales, ADI collapses to ~1 and every SKU looks smooth. Reindex to a full weekly grid first.
CV² on all data instead of non-zero sizes. CV² must use only the positive demand sizes; including zeros distorts it.
Expecting point accuracy to look good. Intermittent point forecasts are inherently "wrong" most weeks (you predict a rate, demand is 0 or a spike). Judge by inventory outcomes, not week-by-week hits.

Deliverables

  • A classification report: the ADI–CV² scatter with quadrant counts for the catalogue.
  • A routing function mapping each SKU to its method, and forecasts produced per route.
  • The obsolescence plot (SBA vs TSB on a dying item) with a one-line takeaway.
Self-check
  • ☐ Are zero-demand weeks explicit in your data?
  • ☐ Can you state what fraction of SKUs are lumpy/intermittent?
  • ☐ Does your plot clearly show TSB decaying and SBA not?

Project 4 · Advanced · ~4.5 hrs

Hierarchical reconciliation across channel & zone

Make your forecasts add up. Build base forecasts at every level of a SKU → channel → zone → national tree, then reconcile them with bottom-up and MinT so the numbers are coherent — and measure whether reconciliation also improves accuracy.

What you'll build

A reconciliation pipeline using Nixtla's hierarchicalforecast: define the aggregation structure, generate base forecasts at every node with statsforecast, then reconcile with BottomUp and MinTrace. You'll verify coherence (children sum to parents to the cent) and compare accuracy at each level — testing the claim that MinT can beat bottom-up even at the top.

Real-world analogue: the S&OP problem — one coherent demand number from SKU to national, across GT/MT/Ecom/IN channels and zones. Dataset: M5 ↗ (state → store → category gives a natural hierarchy). Reinforces: reference Hierarchical reconciliation ↗.

Base forecasts (incoherent) → reconciled (coherent)
1 · Define the hierarchy spec (~40 min)
Choose your levels (e.g. state → store → category). Use aggregate() to build the series at every level and the summing matrix S that encodes the tree.
2 · Generate base forecasts everywhere (~60 min)
Fit AutoETS/AutoARIMA on every node (top, middle, leaves) with statsforecast. These are deliberately independent, so they won't be coherent.
3 · Verify the incoherence (~20 min)
Sum the leaf forecasts and compare to the parent forecast — confirm they differ. This is the problem reconciliation solves.
4 · Reconcile (BU + MinT) (~50 min)
Apply BottomUp and MinTrace(mint_shrink). Confirm children now sum to parents exactly at every level.
5 · Evaluate accuracy by level (~50 min)
Backtest base vs BU vs MinT with rolling-origin; report WMAPE at national, zone, and leaf levels. Look for where MinT beats BU.
6 · Interpret & write up (~30 min)
Explain the accuracy/coherence trade-offs you observed and which method you'd ship for an S&OP process.

Build the hierarchy

from hierarchicalforecast.utils import aggregate # each list is one level of the tree, coarse to fine spec = [ ["state"], ["state", "store"], ["state", "store", "cat"], ] Y_df, S_df, tags = aggregate(df, spec) # Y_df has every level stacked; S_df is the summing matrix

Base forecasts at every node

from statsforecast import StatsForecast from statsforecast.models import AutoETS sf = StatsForecast(models=[AutoETS(season_length=7)], freq="D", n_jobs=-1) Y_hat = sf.forecast(df=Y_df, h=28, fitted=True) Y_fitted = sf.forecast_fitted_values() # needed by MinT to estimate error covariance

Reconcile with BottomUp and MinT

from hierarchicalforecast.core import HierarchicalReconciliation from hierarchicalforecast.methods import BottomUp, MinTrace hrec = HierarchicalReconciliation(reconcilers=[ BottomUp(), MinTrace(method="mint_shrink"), ]) Y_rec = hrec.reconcile(Y_hat_df=Y_hat, Y_df=Y_fitted, S=S_df, tags=tags) # Y_rec now has coherent columns: AutoETS/BottomUp and AutoETS/MinTrace_method-mint_shrink

Prove coherence

# sum of leaf forecasts must equal the national forecast, per method col = "AutoETS/MinTrace_method-mint_shrink" leaves = Y_rec.loc[tags["state/store/cat"]] national = Y_rec.loc[tags["state"]].groupby("ds")[col].sum() assert np.allclose(leaves.groupby("ds")[col].sum(), national) # coherent ✓

Expected results

  • Base (independent) forecasts will be incoherent — leaf sums won't match parents. After BU and MinT, coherence holds exactly (your assert passes).
  • Bottom-up is unbiased but often noisier at the top; MinT frequently matches or beats it at aggregate levels while staying competitive at the leaves — the result the method promises.
  • Improvements are usually modest but consistent; the bigger operational win is coherence itself (a single number that ties out across the org).
  • MinT needs a reasonable error-covariance estimate; mint_shrink is the robust default — plain mint_cov can be unstable on short histories.

Common errors

Forgetting fitted values. MinT needs in-sample fitted values to estimate the error covariance — pass fitted=True and supply them, or MinT can't run.
Inconsistent keys across levels. The aggregation spec must use columns that nest cleanly; a store appearing under two states breaks the tree.
Comparing accuracy only at one level. Reconciliation's effect differs by level — report national, middle, and leaf WMAPE separately.

Deliverables

  • A reconciliation notebook: spec → base forecasts → BU + MinT → coherence proof.
  • A by-level accuracy table (base vs BU vs MinT) from rolling-origin backtesting.
  • A short recommendation: which method you'd use for S&OP and why.
Self-check
  • ☐ Does your coherence assert pass for both BU and MinT?
  • ☐ Can you show a level where MinT beats bottom-up?
  • ☐ Do you understand why independent forecasts were incoherent?

Hard Use Project 2's LightGBM as the base forecaster instead of AutoETS, then reconcile — does a stronger base change which reconciliation method wins?
Project 5 · Advanced–Expert · ~4.5 hrs

Backtest & champion/challenger harness

Tie it all together. Build a reusable rolling-origin harness that scores any model with WMAPE/bias/MASE per horizon and segment, add a foundation-model (Chronos/TimesFM) zero-shot challenger, and implement champion/challenger selection — the system that decides what ships.

What you'll build

A model-agnostic backtesting harness that takes any forecaster, runs rolling-origin CV, and emits per-fold, per-horizon, per-segment WMAPE/bias/MASE with seasonal-naive always included as the floor. You'll plug in the classical baselines (P1), LightGBM (P2), and a foundation-model zero-shot forecaster (Chronos or TimesFM), then build champion/challenger logic with hysteresis that promotes a model only when it wins consistently.

Real-world analogue: the evaluation-and-selection backbone of a production forecasting platform. Dataset: Rossmann ↗ (rich promo/holiday covariates) plus any series from earlier projects. Reinforces: reference Evaluation ↗, Advanced evaluation (Diebold-Mariano) ↗, Probabilistic & conformal ↗, Forecast→decision ↗ and the 2026 frontier ↗.

The harness: every model scored the same way, then selected
1 · Define the harness interface (~40 min)
A backtest(forecaster, df, h, n_windows, step) that loops rolling origins, calls the forecaster on each train slice, and collects predictions tagged with fold and horizon. Every forecaster shares one signature.
2 · Scoring with segments & horizons (~45 min)
Compute WMAPE/bias/MASE overall, by horizon, and by segment (store type, promo vs non-promo). Always include SeasonalNaive as a column.
3 · Wrap existing models (~40 min)
Adapt P1 classical and P2 LightGBM to the harness interface so they're directly comparable.
4 · Add a foundation-model challenger (~55 min)
Wrap Chronos (or TimesFM) zero-shot: feed each series' history, get a quantile forecast, no training. Slot it into the harness like any other model.
5 · Champion/challenger selection (~40 min)
Pick the champion per segment by mean WMAPE across folds; promote a challenger only if it beats the champion by a margin on most folds (hysteresis to avoid flip-flopping).
6 · Report & decide (~30 min)
Produce the scorecard and a clear recommendation: which model wins where (e.g. GBM on covariate-rich promo series, foundation model on cold-start), and what you'd deploy.

The model-agnostic rolling-origin harness

import pandas as pd, numpy as np def backtest(forecaster, df, h=28, n_windows=4, step=28): """forecaster(train_df, h) -> df[unique_id, ds, yhat].""" cutoffs = sorted(df["ds"].unique())[-(n_windows*step + h):][::step][:n_windows] rows = [] for k, cut in enumerate(cutoffs): train = df[df.ds <= cut] test = df[(df.ds > cut) & (df.ds <= cut + pd.Timedelta(days=h))] pred = forecaster(train, h).merge(test, on=["unique_id","ds"]) pred["fold"] = k pred["horizon"] = (pred.ds - cut).dt.days rows.append(pred) return pd.concat(rows, ignore_index=True)

Score by horizon and segment

def wmape(g): return (g.yhat - g.y).abs().sum() / g.y.abs().sum() def scorecard(bt, by=None): keys = ([by] if by else []) out = (bt.groupby(keys + ["fold"]).apply(wmape).rename("wmape").reset_index()) return out.groupby(keys)["wmape"].agg(["mean", "std"]) if keys else out.wmape.agg(["mean","std"]) print(scorecard(bt_lgb, by="horizon")) # degradation curve print(scorecard(bt_lgb, by="promo")) # promo vs non-promo

Foundation-model zero-shot challenger (Chronos)

import torch, numpy as np, pandas as pd from chronos import ChronosPipeline pipe = ChronosPipeline.from_pretrained("amazon/chronos-bolt-base", device_map="cuda") def chronos_forecaster(train, h): out = [] for uid, g in train.groupby("unique_id"): ctx = torch.tensor(g.sort_values("ds")["y"].values, dtype=torch.float32) q, mean = pipe.predict_quantiles(context=ctx, prediction_length=h, quantile_levels=[0.5]) future = pd.date_range(g.ds.max() + pd.Timedelta(days=1), periods=h) out.append(pd.DataFrame({"unique_id":uid, "ds":future, "yhat": q[0,:,0].numpy().clip(min=0)})) return pd.concat(out, ignore_index=True)

Champion / challenger with hysteresis

def select(scores, champion, margin=0.02, min_win_folds=3): """scores: {model: wmape_per_fold array}. Promote only on consistent wins.""" champ = scores[champion] best, best_mean = champion, champ.mean() for m, s in scores.items(): wins = (s < champ).sum() if s.mean() < best_mean - margin and wins >= min_win_folds: best, best_mean = m, s.mean() # beat by margin AND on most folds return best

Expected results

  • On Rossmann's covariate-rich, promo-heavy series, the tuned LightGBM usually beats the zero-shot foundation model — the honest 2026 result. The gap narrows or reverses on short-history/cold-start series.
  • The foundation model is a strong, near-instant baseline — often beating classical methods with zero training, which is genuinely useful for cold-start.
  • WMAPE rises with horizon for all models; segment slices reveal promo weeks are where models differ most.
  • The selector should not flip the champion on a single lucky fold — that's the hysteresis working.

Common errors

Recomputing features outside the fold. The harness must build features inside each train slice, or you reintroduce leakage. Keep feature logic behind the forecaster interface.
Comparing models on different folds. Every model must see identical train/test splits, or the comparison is meaningless. The harness enforces this by construction.
Judging the foundation model unfairly. Zero-shot can't see your promo covariates; note that when interpreting — it's a covariate handicap, not pure model weakness.
GPU/memory for foundation models. Batch series and use the Bolt variant; running thousands of series one-by-one on CPU is slow.

Deliverables

  • A reusable backtest() + scorecard() module that any future model can plug into.
  • A multi-model scorecard (naive, classical, LightGBM, foundation) by horizon and segment.
  • A champion/challenger selector with hysteresis and a written deployment recommendation.
Self-check
  • ☐ Do all models share identical folds and a leakage-free interface?
  • ☐ Can you state where each model wins (cold-start vs covariate-rich)?
  • ☐ Does the selector resist single-fold noise?

Beyond the projects · share it

Publish your work

A forecasting project nobody can read is half-finished. Turn these into portfolio pieces and internal artifacts that demonstrate judgment, not just code.

Notebook structure that reads well

  1. The question & the baseline — state what you're forecasting and the seasonal-naive floor up front.
  2. Data & EDA — a few plots that motivate the method (seasonality, intermittency).
  3. Method & why — the model and the reasoning, not just the call.
  4. Backtest & results — the leaderboard vs naive, by horizon and segment.
  5. Honest limitations — where it fails and what you'd do next.

The write-up template

Lead with the result a planner cares about: "WMAPE improved from X to Y vs the prior baseline; bias fell from +Z% to ~0; here's where it still struggles." Then the how. Recruiters and managers skim — the headline number and an honest limitation section signal seniority more than model complexity does.

Good homes: a clean GitHub repo with a README that shows the leaderboard chart, a short blog post, or an internal one-pager for an S&OP audience.

Pre-publish checklist

☐ Runs top-to-bottom on a fresh kernel from the public dataset.
☐ Every model is compared to seasonal-naive on the same rolling-origin folds.
☐ No leakage — features audited; a reviewer could verify cutoff discipline.
☐ Results reported by horizon and segment, with bias alongside magnitude.
☐ Limitations stated honestly; next steps named.
☐ One chart a non-technical stakeholder can read in five seconds.
